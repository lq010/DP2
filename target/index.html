4#Design#an#implementation#
The servicer allow add and remove NFFGS, these operation can
be executed on /nffgs and/nffgs/{nffgName} resource. The XML
schema is as similar as the previously designed for assignment 1,
but the policies element is removed from NFFG, a new root
element TaggedPolicies is added.
The service allows add multiple NFFGs in one quest. If an
error(the NFFG already loaded) occurs, the service will still add
new NFFGs, and return a error message with the list of error
NFFGs. Execute the verification of policies is action, If the
policy is already in the service, just treat the verificationResult
like a resource, but if the service just verify the policy passed by
client, treat the verification as a resource. If verify the policies
passed by the client, the element belongs must occurs, and its
sub-element nffgs rel attribute is required.
The service has to manage concurrency, so some methods are
synchronized. Methods about add NFFGs and management
policies used two different static objects lock because of
remove NFFGs operation and verify policies passed by client
operation are disabled, in this case, the service can add NFFGs
and add or update policies at the same time. Of course the policy
cannot be added if it refers to a not existing NFFG.
